2001-06-13 : version 0.4 (database model has changed) - Francois Bloque alias PSG version (hence the bugs :) )
  • Adding a new part of the deal module : the ParentDeal manager
    • you can group deals under a parentdeal
    • you can create parentdeals
    • you can search for parentdeals
    • from a deal's page, you can affect the deal to a parentdeal
    • from a deal's page, you can consult information about its parentdeal
    • if all the deals of a parentdeal are archived, then the parentdeal is archived.
    • from a parentdeal's page, you can create a deal which will be automatically attached to the parentdeal, and whose default managers will be the ones of the parentdeal.
    • Only an editor or an admin can affect deals to a parentdeal
    • The status of a parentdeal is the less advanced status among all its deals.
    • The amounts of a parentdeal are the sums of the proposal amounts of its deals (determined by a balance "plus" or "minus" see below).
  • The Computer module can be accessed from all themes now
  • Deal_proposal, Deal_origin_id and Deal_manager_id are not used anymore.
  • The Zipcode is now 14 characters
  • The Deal label is now 128 characters
  • Deal Status have an order now.
  • Deal Types have a balance : plus or minus.
  • Deal types, status and categories are all customizable (Deal submenu "Administration").
  • New fields : deal_number and company_number which can be used as registration numbers (they can be blank), and the character string "deal_todo" which is the next thing to do with the deal.
  • Save and restore of the user's preferences (lang, theme, etc. ) when he quits or logs in obm.
  • Default preferences profile inserted for a new user
  • Deletion of all preferences of a user when he is removed from obm
  • A company has a state : "internal" or "external" ("internal" means it's the company of the user, it's "my company"). It replaces the company of id 1. Then the "internal" contacts are the ones who belong to an internal company.
  • Possibility to protect your own data from the other users by specifying deal or contact as "private". Only the creator of the deal or the contact (the one who inserted data) can see these items in the lists, access their detailled page and set them to public or private.
  • Direct access (link) to the last visited deal, the last visited contact, and the last visited company.
  • All list displays (contacts, companies, deals, parentdeals) are managed by the OBM_DISPLAY class, which allow to order a list by one or two columns and to customize all display options.
  • The contacts, parentdeals, or companies deletion are checked : we can't delete companies or contacts linked to deals, or parentdeals which contain deals.
  • Better security : a not allowed user can't access a forbidden section by using the url anymore.
  • Documentation : OBM-PREFERENCES-HOWTO and OBM-DISPLAY-LISTS-HOWTO
  • Documentation : New developement guidelines for query functions and html display functions (only in the Deal section for the moment - see dev-guidelines.html for more details)
