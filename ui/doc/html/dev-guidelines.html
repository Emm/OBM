OBM Development Guidelines

Coding standards now mandatory to harmonize the OBM code
Version 0.6.6 - 2003-07-17
Pierre Baudracco

Introduction

As OBM is evolving towards a more ambitious and complex application, we need more rigorous coding standards.

This document describes what is now mandatory coding rules for OBM code.

OBM Files

File Size and global presentation

OBM is a web application and so is deployed on Internet/Intranet servers. These servers are often accessed from distant computers (via telnet, ssh,...) so OBM code has to be readable from a terminal in 80 col mode.

OBM line separators (comments, headers,...) should always be 79 chars long and never more.

Indentation should be 2 characters steps. 

///////////////////////////////////////////////////////////////////////////////

Each truncated line should be so with this constraint in mind. 

  $req = "select Company.*, company_id as Id, companytype_label
          from Company, CompanyType 
          where company_type_id=companytype_id";

Of course some lines should not be truncated (function definitions, ...)

HTML should always be writen in lower caps letters (yes there is a big update here to do !)
HTML output should always be separated from process logic and should only contain simple vars.

Module Structure

OBM is divided into application sections and modules. A section beeing a set of modules.

Each module must be divided into 4 files :

  • (1) module_index.php
  • (2) module_display.inc
  • (3) module_query.inc
  • (4) module_js.inc

Module file Description
module_index.php Main module file, is the module skeleton which calls and coordinates the functions contained in all the other files.
The $action variable dictates the module execution.
module_display.php Display module file which contains display functions
module_query.php Query module file which contains query functions (database access)
module_js.php Javascript module file which contains javascript functions.

The module index file : module_index.php

Each module index file must have an header, referencing all actions implemented, as described below : 

///////////////////////////////////////////////////////////////////////////////
// OBM - File  : company_index.php                                           //
//     - Desc  : Company Index File                                          //
// 1999-03-19 Pierre Baudracco                                               //
///////////////////////////////////////////////////////////////////////////////
// $Id$ //
///////////////////////////////////////////////////////////////////////////////
// Actions           -- Parameter
// - index (default) -- search fields  -- show the company search form
// - search          -- search fields  -- show the result set of search
// - new             --                -- show the new company form
// - detailconsult   -- $param_company -- show the company detail
// - detailupdate    -- $param_company -- show the company detail form
// - insert          -- form fields    -- insert the company
// - update          -- form fields    -- update the company
// - check_delete    -- $param_company -- check links before delete
// - delete          -- $hd_company_id -- delete the company
// - admin           --                -- admin index (kind)
// - kind_insert     -- form fields    -- insert the kind
// - kind_update     -- form fields    -- update the kind
// - kind_checklink  --                -- check if kind is used
// - kind_delete     -- $sel_kind      -- delete the kind
// - display         --                -- display and set display parameters
// - dispref_display --                -- update one field display value
// - dispref_level   --                -- update one field display position 
// External API ---------------------------------------------------------------
// - ext_get_id      -- $title         -- select a company (return id) 
// - ext_get_id_url  -- $url, $title   -- select a company (id), load URL

Names of $action variable should be coherent. Default actions names should be used :

Module_index.php global variable to define

 the_index.php file must define :

variable content
$path relative path to the obm php root(usually .. for a module)
$section Id of the section (COM, PROD,...), module name capitalized
$menu Id of the module (USER, CONTACT,...), module name capitalized
$obminclude get from the ENV var : $OBM_INCLUDE_VAR or default to "obminclude"

Here is a correct module_index.php variable definition : 

$path = "..";
$section = "COM";
$menu = "COMPANY";
$obminclude = getenv("OBM_INCLUDE_VAR");
if ($obminclude == "") $obminclude = "obminclude";

Included files : how to, order

There is 3 global files which must be included in all OBM module :

  • (1) obmlib.inc
  • (2) global.inc
  • (3) global_pref.inc

This order must be respected in all module.

Global file Description
obmlib.inc This file define the library objects like OBM_Sess, or OBM_DB. It must be included before the call of page_open() in any case.
global.inc This file define the global variables. It loads two other includes : global_display.inc and global_query.inc. In the settings module and in obm.php it must be included before the call of page_open(). In the other case it could be defined before or after page_open(). For more homogenity we decided to include it before the call of page_open() in any case.
global_pref.inc This file sets up the global settings. It use some session variable so it must be included after the call of page_open() in any case.

Here is a correct module_index.php includes bloc : 

$path = "..";
$menu = "COMPANY";
$obminclude = getenv("OBM_INCLUDE_VAR");
if ($obminclude == "") $obminclude = "obminclude";
require("$obminclude/phplib/obmlib.inc");
include("$obminclude/global.inc");
page_open(array("sess" => "OBM_Session", "auth" => "OBM_Challenge_Auth", "perm" => "OBM_Perm"));
$perm->check("user");
include("$obminclude/global_pref.inc");

require("company_query.inc");
require("company_display.inc");

Parameters handling

No parameter should be used "as is" or as passed. For each module the function get_param_module() is in charge of collecting the parameters and assigning these to the $module associative array which is then be used all over the module to get the values.

This allows to change easily forms widget names in the templates without numerous modifications on the code. 

///////////////////////////////////////////////////////////////////////////////
// Stores Company parameters transmited in $company hash
// returns : $company hash with parameters set
///////////////////////////////////////////////////////////////////////////////
function get_param_company() {
  global $tf_num, $cb_state, $tf_name, $sel_kind, $tf_ad1, $tf_ad2, $tf_zip;
  ...
  global $cdg_param;

  if (isset ($param_company)) $company["id"] = $param_company;
  if (isset ($tf_num)) $company["num"] = $tf_num;
  if (isset ($cb_state)) $company["state"] = $cb_state;
  if (isset ($tf_name)) $company["name"] = $tf_name;
  ...

  if (debug_level_isset($cdg_param)) {
    if ( $company ) {
      while ( list( $key, $val ) = each( $company ) ) {
        echo "
company[$key]=$val";
      }
    }
  }

  return $company;
}

With php set to magic_quotes_gpc = On, parameters strings are automatically backslashed (quotes).

All the variable formatting must be done here, but there is an exception : some code parts used backslashed var form (ex: in queries) and other not (eg: when value must be displayed in a text field). So some "stripslashes()" calls will remains in module_display.inc functions.

Form data handling

A generic module offers forms to create and update an entity. These forms are handled by a unique function (html_module_form()) which act depending on paratemers given. Depending on the action, default parameters can be set, but can be superceded if values are given.

///////////////////////////////////////////////////////////////////////////////
function html_company_form($action, $co_q, $kind_q, $act_q, $usr_q, $company) {
  ....

  // if update mode and first display values are taken from database
  if ($action == "detailupdate") {
    $id = $co_q->f("company_id");
    $num = $co_q->f("company_number");
    $archive = ($co_q->f("company_archive") == 1 ? " checked" : "");
    $name = $co_q->f("company_name");
    $kind = $co_q->f("company_type_id");

  }

  // If parameters have been given, they supercede the default action value

Functions

Each function must have an header, referencing all parameters and the return value (if any).
Parameters must be described after the line 

// Parameters:

Returned value must be described after the line 

// Returns:

Here is a correct function header : 

///////////////////////////////////////////////////////////////////////////////
// Company Form Data checking and formatting                                 //
// Parameters:
//   - $cid       : company id
//   - $company[] : values checked
//     keys used  : num, name, zip, phone, fax, web, email
// Returns:
//   - (true | false) : true if data are ok, else false

  • Display functions:
    • All variables needed by the display function (other than labels, colors, or session variables) must be transmitted as arguments.
    • The name of the function is like "dis_module_$$$" (ex. dis_deal_search_form() ).
    • The file has to be called in the _index.php file with an "include" and not a "require".

  • Query functions:
    • The name of the function must be like "run_query_$$$" where "$$$" represent the action performed.
    • All variables needed by the query (like label or id) have to be transmitted as arguments.
    • The OBM_DB object which execute the query is local to the function.
    • The function return the result of the query.
    • The file has to be called in the _index.php file with an "include" and not a "require".

Each user in OBM can set individually its debug level. The user debug level is stored in the session variable $set_debug. The debug level is a combination of different debug flags or switches.

Debug flags defined

For 0.6.2, OBM defines 4 independant debug switches.

Debug functions API

The $set_debug session variable contains the user current debug level which should be tested against the various switches to display debug information.

OBM defines these functions (in global.inc) to help test and display debug info.

debug_level_isset()

///////////////////////////////////////////////////////////////////////////////
// Test if the debug flag is set
// Parameters:
//   - $debug : debug flag
// Returns:
//   - true || false : true is flag set
///////////////////////////////////////////////////////////////////////////////
function debug_level_isset($debug) {
  global $set_debug;

  if (($set_debug > 0) && (($set_debug & $debug) == $debug)) {
    return true;
  } else {
    return false;
  }
}

debug_level_isset($debug_flag) returns true if the flag is set for the user;

Example of use 
if (debug_level_isset($cdg_param)) {

display_debug_msg()

///////////////////////////////////////////////////////////////////////////////
// Display a debug message
// Parameters:
//   - $msg         : message to display
//   - $debug (opt) : debug flag required
///////////////////////////////////////////////////////////////////////////////
function display_debug_msg($msg, $debug=0) {
  global $set_debug;

  if (($debug > 0) && (($set_debug & $debug) == $debug)) {
    echo $msg;
    echo "
\n";
  }
}

Date handling

OBM allow different date representation depending on the user current language.
Each user in OBM can individually set its language preference.

To allow database independance, OBM works with Unix timestamps that the database must provide for dates.

Database storage

Date should be store with date data type fields in database.
Eg: Date for simple date or timestamp for MySQL.

The only constraint is : Select query must return date in Unix Timestamp format (which is used by OBM and PHP).

Insert or Update must provide date in ISO format (AAAA-MM-JJ or AAAA-MM-JJ hh:mm:ss).

Developper Requirements

Dates can be stored in the prefered database format but must be converted (if needed) when retrieved from the database to Unix timestamp format. This can lead to different queries for different database.

So OBM always get UNIX Timestamp date format and can display dates according to user prefences. You can modify date display representation in OBM simply by selecting the format in the user settings screen.

OBM provides some date formating functions to handle user preferences.

date_format($timestamp)

Displays the date corresponding to the timestamp in the user selected format (date).

datetime_format($timestamp)

Displays the date-time corresponding to the timestamp in the user selected format. (date + time).

isodate_format($timestamp)

Displays the date corresponding to the timestamp in the iso format (date). In forms fields, date are displayed in ISO format.

Date format provided by OBM

This could be extended in the settings module.

Constant Value Format / Language example
$cda_iso Y-m-d ISO AAAA-MM-JJ
$cda_en m/d/Y English MM-DD-AAAA
$cda_fr d/m/Y French DD/MM/AAAA
$cda_txt d M Y Textual 25 Jan 2003

Regular Expressions

PHP provides 2 kind of regular expression :

Kind Function prefix Pattern syntax OBM use
PCRE (Perl compatible regular expression) preg_ "/^[a-z]$/" Yes
Regexps (Posix extended) ereg_ ou eregi_ "^[a-z]$" No

In OBM for performance and functionality, only PCRE regexps must be use.

URL handling

OBM used to call $sess->url() for eah url invocation in the code. Now each url construction must call url_prepare() which handle all operations an url should received.

It will ease transition if we decide to migrate from the current session object ($sess) to another model.

Operations handled by url_prepare() :

  • Call $sess->url() to allow use without cookies
  • Replace '%' with '%25' to avoid URL interpreting %hh as special character (ex: searching %aa, lead to a special character when appearing in the url)

Variables Names

Variables names must be explicit without beeing exhaustive.
Ex: $param_theme. No $a, $i, $tp neither $parameter_theme_for_settings_screen.

If more than one word is to be used in a variable name the separator "_" should be used.
